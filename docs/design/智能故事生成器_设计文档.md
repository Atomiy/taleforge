**Agent 协作面板交互设计（详细）：**

- **状态可视化**：

### 3.3 多智能体协作引擎详细设计

本项目核心创新在于**多智能体分工协作**，替代传统的单一 LLM 调用模式。各智能体通过协调器（Orchestrator）串联，以结构化数据（JSON）作为通信载体。

#### 3.3.1 协调器（Orchestrator）

协调器是整个多智能体系统的"导演"，负责：

- 接收前端请求，解析用户参数
- 按序调度各 Agent 执行（Planner → Character → Writer → Polisher）
- 管理 Agent 间的数据传递（将上游 Agent 的输出作为下游 Agent 的输入）
- 向前端推送各 Agent 的实时状态（SSE 事件流）
- 汇总最终结果，保存历史记录

```python
# 协调器核心流程示意
class StoryOrchestrator:
    async def generate_story(self, request: StoryRequest) -> AsyncIterator[str]:
        # 1. 策划Agent + 角色Agent 并行执行（无强依赖，缩短约20%-30%时间）
        yield json.dumps({"agent": "planner", "status": "running"})
        yield json.dumps({"agent": "character", "status": "running"})
        
        outline_task = self.planner_agent.run({"request": request})
        character_task = self.character_agent.run({"request": request})
        
        outline, characters = await asyncio.gather(outline_task, character_task)
        
        yield json.dumps({"agent": "planner", "status": "done", "data": outline})
        yield json.dumps({"agent": "character", "status": "done", "data": characters})
        
        # 2. 写作Agent生成初稿（流式）
        yield json.dumps({"agent": "writer", "status": "running"})
        async for chunk in self.writer_agent.run_streaming({
            "outline": outline, "characters": characters, "request": request
        }):
            yield json.dumps({"agent": "writer", "status": "writing", "data": chunk})
        
        draft = self.writer_agent.get_full_draft()
        
        # 3. 润色Agent润色（含一致性校验）
        yield json.dumps({"agent": "polisher", "status": "running"})
        final = await self.polisher_agent.run({
            "draft": draft, "outline": outline, 
            "characters": characters, "request": request
        })
        yield json.dumps({"agent": "polisher", "status": "done", "data": final})
        
        yield json.dumps({"agent": "orchestrator", "status": "complete"})
```

#### 3.3.2 策划智能体（Planner Agent）

**职责：** 根据用户输入的主题、体裁、风格、角色，生成完整的故事大纲。

**输入：** 用户请求参数（主题、体裁、风格、目标字数、角色初步设定）
**输出：** 结构化大纲（JSON），包含：
- 故事标题
- 选用的体裁结构模板（A~H，见第七章）
- 阶段划分（各阶段名称、占比、关键情节节点）
- 冲突类型建议
- 章节列表（如适用）

**模型参数：** temperature = 0.5（偏向逻辑和结构化）

#### 3.3.3 角色智能体（Character Agent）

**职责：** 
- 如用户提供简略角色，自动扩充为**角色深度模型**（四层：身份、驱动、矛盾、弧线）
- 在写作过程中**维护角色一致性**（检查角色言行是否符合设定）

**输入：** 用户角色设定 + 策划Agent生成的大纲
**输出：** 完整的角色设定表（JSON），包含角色在各阶段的状态变化

**模型参数：** temperature = 0.5（需要稳定和一致性）

#### 3.3.4 写作智能体（Writer Agent）

**职责：** 根据大纲和角色设定，生成故事正文。

**输入：** 大纲（JSON）+ 角色设定表（JSON）+ 用户风格要求
**输出：** 故事初稿（Markdown 格式）

**特性：**
- 支持**分段/分章节并行写作**（大纲有多个章节时，可并发调用多个 Writer 实例）
- 支持**流式输出**（SSE 逐字推送至前端）
- 严格遵循大纲中的阶段占比和情节节点

**模型参数：** temperature = 0.8（创意优先）

#### 3.3.5 润色智能体（Polisher Agent）

**职责：** 对写作Agent生成的初稿进行质量提升。

**输入：** 故事初稿 + 风格要求
**输出：** 润色后的最终故事（Markdown 格式）

**润色任务：**
- 语言润色（修辞、句式多样性）
- 错别字和语病纠正
- 风格统一（确保全文符合选定的风格）
- 连贯性增强（段落衔接、过渡自然）
- **闭环一致性校验**：传入角色设定表 + 前情摘要，检查全文是否有逻辑矛盾或设定偏离（如角色左腿受伤后却健步如飞）
- 角色一致性复核（最终把关）

**模型参数：** temperature = 0.3（精确优先）

#### 3.3.6 配图智能体（Image Agent）—— 可选扩展

**职责：** 为故事关键场景生成配图，增强故事的视觉表现力。

**触发条件：** 用户在创作页勾选"为故事生成配图"选项，且系统检测到本地或远程 ComfyUI 服务可用。

**工作流程：**

```
润色后的最终故事
    │
    ▼
┌─────────────────────────────────┐
│  1. 场景提取                      │
│     - 从故事中识别 1-3 个关键场景   │
│     - 提取场景的文字描述（50字内）  │
└──────────────┬──────────────────┘
               │ 场景描述文本
               ▼
┌─────────────────────────────────┐
│  2. Prompt翻译与优化              │
│     - 将中文场景描述翻译为英文      │
│     - 追加风格标签（画风/光线等）   │
└──────────────┬──────────────────┘
               │ 英文Prompt
               ▼
┌─────────────────────────────────┐
│  3. 调用ComfyUI API              │
│     - POST /prompt 提交工作流JSON  │
│     - 轮询GET /history获取结果    │
│     - 获取生成图片的文件路径       │
└──────────────┬──────────────────┘
               │ 图片URL
               ▼
┌─────────────────────────────────┐
│  4. 返回前端展示                  │
│     - 故事卡片旁展示配图缩略图      │
│     - 点击可查看大图               │
└─────────────────────────────────┘
```

**ComfyUI 接口设计：**

ComfyUI 提供基于 HTTP 的 API 接口，本项目通过 Python `requests` 库调用：

| 接口 | 方法 | 用途 |
|------|------|------|
| `/prompt` | POST | 提交工作流执行请求（传入 workflow JSON + prompt） |
| `/history` | GET | 查询历史任务记录，获取生成结果（含图片文件路径） |
| `/queue` | GET | 查看当前队列状态 |
| `/ws` | WebSocket | 实时接收执行进度（可选，用于前端进度条） |

**工作流配置：**

- 系统预置一套 **ComfyUI 工作流 JSON**（存储在 `backend/data/comfyui_workflow.json`），包含：
  - 文生图基础节点（CLIP Text Encode → KSampler → VAE Decode → Save Image）
  - 固定的模型和参数配置（避免每次调用都重新配置）
- 运行时动态替换 workflow JSON 中的 `prompt` 字段为场景描述

**ComfyUI 连接配置：**

```python
# config.py
COMFYUI_ENABLED = True  # 是否启用配图功能
COMFYUI_API_URL = "http://127.0.0.1:8188"  # ComfyUI 服务地址
COMFYUI_WORKFLOW_PATH = "data/comfyui_workflow.json"  # 预置工作流
```

**模型参数：** 不涉及 LLM temperature，由 ComfyUI 工作流内的 KSampler 参数控制

**降级策略：**
- 若 ComfyUI 服务不可连接（ConnectionRefused / 超时），Image Agent 返回空结果，前端不展示配图区域，不阻塞主流程
- 若场景提取失败（如故事过短无明确场景），跳过配图生成

#### 3.3.7 LLM 调用层（LLM Client）

- 封装 DeepSeek API 调用，所有 Agent 共用同一 Client
- 支持流式（SSE）和非流式两种模式
- 每个 Agent 可独立配置 temperature、max_tokens 等参数
- 统一的错误处理：网络超时、API 限流、Key 无效等

#### 3.3.8 故事后处理器（Story Post-processor）

- 解析最终 Markdown 格式文本
- 提取章节标题、对话段落、叙述段落
- 清理特殊标记和多余空行
- 生成故事摘要（用于历史列表展示）

## 四、技术选型

### 4.1 技术栈总览

| 层级 | 技术选型 | 版本/说明 | 选型理由 |
|------|----------|----------|----------|
| 后端框架 | FastAPI | 最新稳定版 | 异步支持好，自动 API 文档，与作业03技术栈一致 |
| 后端运行 | Uvicorn | 最新稳定版 | ASGI 服务器，性能高 |
| LLM 调用 | requests + SSE | — | 直接调用 REST API，无需额外依赖，可控性强 |
| **Agent通信** | **Python异步函数调用（async/await）** | — | **短学期项目轻量化方案，Agent间通过结构化JSON直接传递，无需消息队列** |
| 数据存储 | JSON 文件 | — | 故事历史用 JSON 存，无需数据库，部署简单 |
| 前端框架 | Vue 3 (CDN) | 3.x | 无需构建工具，直接 HTML 引入，组员零配置运行 |
| 前端样式 | Tailwind CSS (CDN) | 3.x | 原子化 CSS，开发快，无需额外 CSS 文件维护 |
| 代码规范 | PEP 8 + 类型注解 | — | Python 标准规范，IDE 友好 |

### 4.2 Agent 通信方式选型说明

| 方案 | 说明 | 采用与否 | 理由 |
|------|------|----------|------|
| 直接函数调用（async/await） | Agent 之间直接调用 Python 异步函数，通过参数传递结构化数据 | **✅ 采用** | 最简单、最轻量，无额外依赖，适合短学期项目 |
| 消息队列（RabbitMQ/Redis） | Agent 通过消息队列异步通信 | ❌ 不采用 | 引入额外服务，增加部署复杂度，超出项目周期 |
| Celery 任务队列 | 分布式任务调度 | ❌ 不采用 | 需要 Redis 作为 Broker，配置复杂 |

> **设计决策：** 本项目 Agent 之间是**紧耦合的流水线协作**（上游输出直接作为下游输入），而非松耦合的独立服务。因此直接函数调用是最合理的方案，同时保留后续扩展为消息队列的可能性（通过抽象接口）。

### 4.3 不采用某些技术的理由

| 未采用的技术 | 理由 |
|-------------|------|
| LangChain / LangGraph | 引入 20+ 依赖包，抽象层过重；本项目 Agent 逻辑简单明确，直接手写更可控 |
| 数据库（SQLite/MySQL） | 故事历史结构简单（标题+内容+时间），JSON 文件完全够用，省去数据库迁移和配置 |
| React / Vue CLI 构建 | 需要 node_modules + 构建步骤，增加部署复杂度；CDN 引入 Vue 3 即可满足需求 |
| Django / Flask | Django 过重；Flask 同步模型，流式输出实现不如 FastAPI 简洁 |

### 4.4 大模型 API 选择

| 模型 | 用途 | 优势 |
|------|------|------|
| DeepSeek-V3 / DeepSeek-Chat | 主力（所有Agent共用） | 中文能力强，价格低，接口稳定 |
| 阿里通义千问 | 备选 / 对比测试 | 国内可用性好，长文本支持好 |

### 4.5 图片生成服务选择

| 服务 | 用途 | 优势 | 采用与否 |
|------|------|------|----------|
| **ComfyUI** | 故事配图生成（可选） | 开源可控，工作流灵活，本地部署无额外费用，用户已有经验 | **✅ 采用** |
| DALL-E 3 / Midjourney API | 备选图片生成 | 质量高，但需额外付费/网络条件 | ❌ 不采用 |
| Stable Diffusion WebUI | 备选图片生成 | 与ComfyUI功能重叠，但API不如ComfyUI灵活 | ❌ 不采用 |

> **设计决策：** 团队成员已具备 ComfyUI 使用经验，且 ComfyUI 的 `/prompt` + `/history` API 模式非常适合程序自动化调用。配图功能为可选扩展，ComfyUI 服务由用户自行启动（非项目内置），项目仅通过 HTTP API 与其通信。

## 五、系统架构

### 5.1 整体架构

采用**前后端分离 + 多智能体协作**的架构：

- **前端**：Vue 3 + Tailwind CSS，纯静态文件，浏览器直接运行
- **后端**：FastAPI 提供 REST API，内部通过**协调器（Orchestrator）**调度多智能体
- **多智能体层**：4个核心 Agent 分工协作，通过结构化 JSON 数据传递
- **外部服务**：DeepSeek API（大语言模型）

### 5.2 架构分层

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              前端层 (Frontend)                            │
│  ┌─────────────┐  ┌─────────────────┐  ┌─────────────┐  ┌─────────────┐ │
│  │  创作页面    │  │  Agent协作面板   │  │  历史记录页  │  │  设置页面    │ │
│  └──────┬──────┘  └─────────────────┘  └──────┬──────┘  └──────┬──────┘ │
│         └─────────────────────────────────────┴─────────────────┘        │
│                              Vue 3 + Tailwind                            │
└───────────────────────────────────┬─────────────────────────────────────┘
                                    │ HTTP / SSE
```

## 六、API Key 安全方案

### 6.1 风险分析

> ⚠️ **已发生的安全事件：** 在之前的作业03项目中，DeepSeek API Key 被以明文形式存储在 `.env` 文件中，且该文件被包含在作业提交材料中，存在泄露风险。

### 6.2 安全方案设计

#### 方案 A：运行时输入（推荐，用于演示和开发）

| 措施 | 说明 |
|------|------|
| 不在任何文件中硬编码 Key | `.env` 文件只保留空模板，不提交真实 Key |
| Web 设置页面输入 | 用户首次使用时，在 Web 界面输入 API Key，保存在浏览器 LocalStorage |
| 前端传递 Key | 每次请求将 Key 放在请求头中发给后端 |
| 后端不持久化 Key | 后端只接收 Key 用于当前请求，不写入文件或日志 |
| **各Agent共用Key** | **所有Agent通过统一的LLM Client调用，Key由协调器注入** |

> **优点：** 代码仓库中完全不含 Key，任何人克隆仓库都能安全运行。
> **缺点：** 每次打开新浏览器需重新输入 Key（可用 LocalStorage 缓解）。

#### 方案 B：环境变量注入（推荐，用于生产部署）

| 措施 | 说明 |
|------|------|
| 系统环境变量 | 通过操作系统环境变量注入 Key：`export DEEPSEEK_API_KEY=sk-xxxx` |
| 启动脚本加载 | 提供 `start.bat` / `start.sh`，从外部配置文件读取 Key 后启动服务 |
| 配置文件排除 | 在 `.gitignore` 中排除 `.env` 和任何含 Key 的文件 |

### 6.3 具体实现

```bash
# .env.example（提交到仓库的模板，不含真实 Key）
DEEPSEEK_API_KEY=your_api_key_here

# .gitignore（必须包含）
.env
__pycache__/
*.pyc
node_modules/
dist/
```

```python
# 后端读取逻辑（config.py）
import os

API_KEY = os.environ.get("DEEPSEEK_API_KEY", "")
if not API_KEY:
    # 尝试从启动参数读取
    import sys
    if len(sys.argv) > 1:
        API_KEY = sys.argv[1]

# 所有Agent共用同一个LLM Client，统一注入Key
llm_client = LLMClient(api_key=API_KEY)
planner = PlannerAgent(llm_client)
character = CharacterAgent(llm_client)
writer = WriterAgent(llm_client)
polisher = PolisherAgent(llm_client)
```

> 🔴 **紧急操作：** 请立即登录 DeepSeek 开放平台，删除已泄露的 Key（`sk-5307...`），重新生成新 Key。

## 七、故事生成逻辑设计

### 7.1 核心问题

直接让单一 LLM "写一个关于 X 的故事"，往往会产生：

- 情节平淡、缺乏冲突
- 角色脸谱化、没有成长弧线
- 结构松散、节奏拖沓
- 风格前后不一致

**多智能体方案的优势：**

- **专业分工：** 每个Agent专注一个创作环节，Prompt 更聚焦，输出质量更高
- **迭代优化：** 润色Agent对初稿进行二次加工，相当于"审校"环节
- **可调试性：** 各Agent输出结构化数据，可独立测试和优化
- **可扩展性：** 新增Agent（如配图Agent、翻译Agent）无需改动现有代码

### 7.2 叙事方法论（三层模型）

#### 第一层：角色深度模型（Character Depth Model）

每个角色包含四个维度，确保角色立体而非脸谱化：

| 维度 | 说明 | 示例 |
|------|------|------|
| 身份层 (Identity) | 基础信息：姓名、年龄、职业、外貌 | 林小满，16岁，高中生，短发 |
| 驱动层 (Drive) | 核心动机：想要什么？害怕什么？ | 渴望被认可，害怕平庸 |
| 矛盾层 (Contradiction) | 内心冲突：表面与内心的落差 | 表面开朗，内心孤独 |
| 弧线层 (Arc) | 成长轨迹：故事前后如何变化 | 从逃避到面对，学会接纳自己 |

#### 第二层：体裁自适应结构引擎（Genre-Adaptive Structure Engine）

> ⚡ **设计修正：** 单一的四阶段结构无法覆盖所有体裁需求。日常短文不需要"高潮与抉择"，诗歌不需要"冲突升级"，剧本需要"幕"而非"阶段"。因此，系统根据用户选择的**体裁自动匹配对应的叙事结构模板**。

##### 体裁结构模板库

| 体裁 | 结构名称 | 核心特征 |
|------|---------|---------------|
| 短篇小说 / 中篇小说 | 冲突驱动四阶段 | 起承转合，有明确的高潮和结局 |
| 日常小短文 / 散文 | 场景切片结构 | 无剧烈冲突，以观察和感受推进 |
| 剧本 / 戏剧 | 三幕结构 | 铺垫→对抗→解决，强调对话和场景 |
| 童话 / 寓言 | 简化英雄之旅 | 出发→考验→归来，带有道德寓意 |
| 悬疑推理 | 悬念结构 | 谜团→调查→反转→解答 |
| 科幻 / 奇幻 | 探索发现结构 | 发现→探索→危机→领悟 |
| 诗歌 | 意象递进结构 | 起兴→展开→收束，强调意象和韵律 |
| 长篇连载（单章） | 章节钩子结构 | 每章有独立小高潮+大悬念 |

##### A. 冲突驱动四阶段 — 适用于 小说

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 引子与触发 | 20% | 介绍角色日常 → 打破平静的事件 → 初步反应 |
| 发展与深入 | 40% | 主动应对 → 揭示深层矛盾 → 关系变化 → 冲突升级 |
| 高潮与抉择 | 25% | 冲突顶点 → 艰难抉择 → 体现成长 |
| 结局与回响 | 15% | 冲突结果 → 状态变化 → 余韵 |

##### B. 场景切片结构 — 适用于 日常小短文 / 散文

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 场景铺陈 | 30% | 描绘一个具体场景（时间、地点、氛围），不急于进入情节 |
| 细节观察 | 35% | 通过感官细节（视觉、听觉、触觉）展现场景的丰富层次 |
| 情感流动 | 25% | 由外及内，从场景细节引发角色的情绪变化或回忆 |
| 意境收束 | 10% | 不追求明确结局，以画面或感悟自然收尾，留下余味 |

> **关键区别：** 日常短文**不强制冲突**。节奏舒缓，重在氛围营造和情感共鸣，而非情节推进。

##### C. 三幕结构 — 适用于 剧本 / 戏剧

| 幕 | 占比 | 叙事任务 |
|----|------|----------|
| 第一幕：铺垫 | 25% | 建立人物关系 → 介绍背景 → 引发事件（催化剂） |
| 第二幕：对抗 | 50% | 主角追求目标 → 遭遇阻碍 → 冲突升级 → 中点转折 → 逼近绝境 |
| 第三幕：解决 | 25% | 高潮对决 → 核心冲突解决 → 人物命运落定 → 场景收束 |

> **关键区别：** 强调**场景切换**和**对话驱动**，每个场景有明确的进入目标、冲突发展和退出钩子。

##### D. 简化英雄之旅 — 适用于 童话 / 寓言

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 平凡世界 | 15% | 主角的日常生活，简单快乐或略有不满 |
| 冒险召唤 | 15% | 一个事件打破平静，主角收到挑战或任务 |
| 考验与盟友 | 40% | 主角踏上旅程，遇到困难，遇到帮助者，学习成长 |
| 最终考验 | 20% | 最大的挑战，主角运用所学战胜困难 |
| 归来与馈赠 | 10% | 带着收获回到日常，故事带有道德寓意 |

##### E. 悬念结构 — 适用于 悬疑推理

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 谜团呈现 | 15% | 一个不可思议的事件发生，抛出核心疑问 |
| 调查推进 | 35% | 收集线索，提出假说，遭遇误导（红鲱鱼） |
| 反转与危机 | 30% | 关键线索推翻原有认知，主角陷入危险 |
| 真相揭露 | 20% | 所有线索串联，真相反转，解答核心疑问 |

##### F. 探索发现结构 — 适用于 科幻 / 奇幻

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 异常发现 | 20% | 主角发现世界中存在异常现象或未知领域 |
| 深入探索 | 35% | 主动调查/进入未知，逐步揭示世界规则，遭遇阻力 |
| 危机爆发 | 25% | 探索触发了不可控后果，世界或主角面临威胁 |
| 领悟与抉择 | 20% | 主角理解真相，在利益和道德间做出选择 |

##### G. 意象递进结构 — 适用于 诗歌

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 起兴 | 20% | 以一个具体意象切入，建立情感基调 |
| 展开 | 50% | 意象层层递进、转化、对比，情感逐渐加深 |
| 收束 | 30% | 意象回归或升华，情感达到顶点或归于宁静 |

> **关键区别：** 诗歌**不依赖情节**，依赖**意象的递进和情感的流动**。要求语言凝练、有韵律感。

##### H. 章节钩子结构 — 适用于 长篇连载（单章）

| 阶段 | 占比 | 叙事任务 |
|------|------|----------|
| 上章衔接 | 10% | 简要回顾或承接上章结尾，让读者快速进入 |
| 本章推进 | 50% | 推进一条或多条支线，制造小冲突和小高潮 |
| 本章高潮 | 25% | 本章内的独立高潮场景（战斗、对话、发现） |
| 悬念钩子 | 15% | 在结尾处抛出新的疑问或危机，驱动读者继续 |

> **关键区别：** 单章是**大故事中的切片**，不要求完整结局，但必须有**独立可读性**和**悬念钩子**。

#### 第三层：冲突触发机制（Conflict Trigger Mechanism）

在需要冲突的体裁中（小说、剧本、童话、悬疑、科幻），系统根据主题自动匹配冲突类型：

| 冲突类型 | 触发模式 | 适用主题 |
|----------|----------|----------|
| 自我对抗 | 角色内心的两种渴望相互矛盾 | 成长、选择、身份认同 |
| 人际冲突 | 角色与重要他人之间的误解或利益冲突 | 友情、亲情、爱情 |
| 环境挑战 | 外部环境（自然、社会、时代）对角色的压迫 | 冒险、生存、变革 |
| 理想与现实 | 角色追求的理想与现实条件之间的落差 | 梦想、奋斗、妥协 |
| 秘密与真相 | 隐藏的真相逐渐揭开，改变角色认知 | 悬疑、推理、揭秘 |

> **注意：** 日常短文、散文、诗歌等体裁**不强制使用冲突**，冲突触发机制仅在需要时激活。

```
4. 增强段落间的连贯性
```

### 7.5 质量保障机制

| 机制 | 说明 |
|------|------|
| **多智能体分工** | 策划→角色→写作→润色，每步专注单一任务，减少单一Prompt的复杂度 |
| 结构化 Prompt | 通过明确的故事阶段划分，强制 LLM 遵循叙事节奏 |
| 角色设定注入 | 将角色四层模型写入 System Prompt，确保角色一致性 |
| 冲突类型匹配 | 根据主题自动选择最适合的冲突类型，避免故事平淡 |
| **润色Agent复核** | 专门Agent负责最终质量把关，相当于"审校"环节 |
| **闭环一致性校验** | **润色Agent传入大纲+角色+前情摘要，主动修正逻辑矛盾（如受伤后健步如飞）** |
| Temperature 差异化 | 策划/角色/润色使用 0.3-0.5（精确），写作使用 0.8（创意） |
| 后处理校验 | 检查输出是否包含阶段结构标记，如缺失则提示用户 |

### 8.3 关键模块接口设计

#### Agent 抽象基类

```python
from abc import ABC, abstractmethod
from typing import Any

class BaseAgent(ABC):
    """所有智能体的抽象基类."""
    
    def __init__(self, llm_client: LLMClient, name: str):
        self.llm = llm_client
        self.name = name
        self.logger = logging.getLogger(f"agent.{name}")
    
    @abstractmethod
    async def run(self, context: dict[str, Any]) -> dict[str, Any]:
        """执行Agent核心任务.
        
        Args:
            context: 上游Agent的输出数据
            
        Returns:
            本Agent的处理结果，以字典形式传递
        """
        pass
```

#### 协调器接口

```python
class StoryOrchestrator:
    """多智能体协调器，负责调度各Agent协作完成故事生成."""

    def __init__(self, llm_client: LLMClient):
        self.planner = PlannerAgent(llm_client)
        self.character = CharacterAgent(llm_client)
        self.writer = WriterAgent(llm_client)
        self.polisher = PolisherAgent(llm_client)

    async def generate_story(
        self,
        request: StoryRequest,
    ) -> AsyncIterator[AgentEvent]:
        """生成故事，返回Agent状态事件流.
        
        Args:
            request: 用户故事生成请求
            
        Yields:
            AgentEvent: 包含agent名称、状态、数据的事件
        """
        # 1. 策划Agent
        yield AgentEvent(agent="planner", status="running")
        outline = await self.planner.run({"request": request})
        yield AgentEvent(agent="planner", status="done", data=outline)
        
        # 2. 角色Agent
        yield AgentEvent(agent="character", status="running")
        characters = await self.character.run({"request": request, "outline": outline})
        yield AgentEvent(agent="character", status="done", data=characters)
        
        # 3. 写作Agent（流式）
        yield AgentEvent(agent="writer", status="running")
        async for chunk in self.writer.run_streaming({
            "request": request,
            "outline": outline,
            "characters": characters,
        }):
            yield AgentEvent(agent="writer", status="writing", data=chunk)
        
        draft = self.writer.get_full_draft()
        
        # 4. 润色Agent
        yield AgentEvent(agent="polisher", status="running")
        final = await self.polisher.run({
            "draft": draft,
            "request": request,
            "characters": characters,
        })
        yield AgentEvent(agent="polisher", status="done", data=final)
        
        # 完成
        yield AgentEvent(agent="orchestrator", status="complete", data=final)
```

本项目采用多智能体架构的核心优势：
