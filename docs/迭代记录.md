## v0.4.0 — 角色出场约束体系 & 事件驱动设计（当前版本）

> 日期：2026-07-10

### 概述

本次迭代在 v0.4.0 卷规划基础之上，进一步构建了**角色出场约束体系**，解决了多角色、多章节故事中角色状态不一致、出场目的不清晰的核心问题。同时从《深渊回响》v34 的实践中提取了**事件驱动设计**方法论，作为智能体提示词的核心指导原则。通过三部作品（深渊回响、星际余晖、苍狼传说）的交叉验证，证明了该体系的通用有效性。

### 角色出场约束体系

#### 三元出场类型
| 类型 | 含义 | 约束 |
|:-----|:-----|:-----|
| **主动** | 角色在本章正常推进剧情 | 需填写出场目的（推进剧情/塑造人物/表达主题/营造氛围） |
| **被动** | 角色在本章可以被提及，但不出声、不行动 | 出场目的限于信息提供/冲突制造/催化事件 |
| **缺席** | 角色在本章完全不出现 | 出场目的为空，关键剧情中不应出现其名 |

#### 数据模型扩展
- `chapter_characters` — 每章出场人物列表（按章节索引）
- `character_appearance_map` — 角色出场表（按角色索引，标注每章的状态）
- `character_arcs` — 角色弧线（每角色在每卷的角色发展目标）
- `chapter_participation` — 每章每角色的参与类型（主动/被动/缺席）
- `chapter_purpose` — 每章每角色的出场目的（标准分类）

#### 预生成校验器
- **文件**：`scripts/precheck.py`
- **校验规则**（C1–C6）：
  - C1：被动角色不得以"推进剧情"为主目的（不含受限子类）
  - C1b：被动角色的受限子类重叠警告
  - C2：缺席角色不应有出场目的
  - C3：缺席角色不应出现在关键剧情中
  - C4：主动角色应填写出场目的
  - C5：被动角色不应在关键剧情中出现主动动词
  - C6：自定义出场目的合法性检查
- **设计原则**：防呆设计，在 Planner 生成前检测配置层矛盾，避免生成出违背约束的章节

### 事件驱动设计原则

从《深渊回响》v34 迭代中总结的方法论：

> 当某角色在特定章节需要被动/缺席时，检查该章的驱动源是否依赖该角色的出场。若是，则替换为外部事件（环境变化、新发现、外部危机），该角色的叙事功能通过他人转述/数据/痕迹实现。

| 维度 | 角色状态驱动（旧） | 事件驱动（v34） |
|:-----|:-------------------|:----------------|
| 核心问题 | "某角色怎么样了？" | "事件在揭示什么？" |
| 被动角色 | 叙事中心（需出场） | 背景状态（可提及可不提） |
| 其他角色行动 | 围绕该角色 | 围绕事件反应 |
| 角色弧线推进 | 单人线推进 | 全员线同时推进 |

## v0.4.0 — 进度锚点约束（补丁）

> 日期：2026-07-10

### 发现的问题
对比《苍狼传说》（因果驱动）和《星际余晖》（时间驱动）的章节连贯性后，发现**情节倒退**问题：第3章标题与第2章重复（"暗流涌动"），且第3章开篇将剧情"倒带"到了第2章中段的状态。

### 根因分析
1. **Planner 层**：续写提示只传了上章最后 500 字，没有标注"这是上章结尾，新章必须从这之后开始"
2. **Writer 层**：过渡规则只约束了"首句衔接"，没有约束"整章定位"

### 修复方案
#### Planner 改动
- 续写提示 `continuation_info` 拆分为"上章开头（300字）"和"**上章结尾（400字）——新章节必须从这点之后开始**"
- 新增"硬约束：进度锚点"小节（4条规则）

#### Writer 改动
- 在 `_build_system_context` 和 `_build_prompt` 中新增"规则7 — 进度锚点"
- 禁止时间回溯/场景重置，结尾必须比开头明显推进

### 验证对比
| 指标 | 修复前 | 修复后 |
|:-----|:------:|:------:|
| 标题 | 暗流涌动（重复） | 裂口之秘（不重复） |
| 起始状态 | 场景重置到第2章中段 | 接续第2章结尾状态 |
| 剧情节拍 | 2个（原地踏步） | 7个（持续推进） |
| 结尾锚点 | 仍在侦察船 | 获得第二信标坐标 |

## v0.4.0 — 卷规划与跨章角色继承（初始版本）

> 日期：2026-07-10

### 概述
实现了**卷级大纲规划系统**，解决了续写时角色状态不跨章传递的核心问题，并完成了第三部作品「深渊回响」的完整三章生成验证。

### Volume 模型扩展
| 新增字段 | 类型 | 说明 |
|:---------|:-----|:-----|
| `outline` | str | 卷故事大纲（含每章简述） |
| `responsibility` | str | 本卷在整部作品中的职责 |
| `connection_to_prev` | str | 与上一卷的衔接说明 |
| `connection_to_next` | str | 对下一卷的铺垫说明 |
| `characters` | List[dict] | 本卷出场人物（name/identity/brief） |
| `foreshadowings` | List[str] | 本卷伏笔清单 |

### StoryRequest 模型扩展
新增字段：`work_id`, `volume_id`, `chapter_number` — 使生成请求能关联到具体作品和卷。

### 卷规划注入生成流程（三层传递）
- **Orchestrator**：启动时根据 `request.work_id` + `request.volume_id` 加载卷规划
- **Planner**：`context["volume_plan"]` — 提示词中标注"卷规划优先级高于所有其他信息"
- **Character**：卷规划人物合并到 `existing_characters`，跳过已存在角色
- **Writer**：通过 `volume_plan` 参数传入，提示词中标注"优先级高于本章大纲"

### 前端卷规划面板
- 每卷新增「章节/规划」tab 切换
- 规划 tab 包含编辑字段：卷职责、卷大纲、卷间衔接、出场人物（可增删行）、伏笔清单（可增删）
- 「保存规划」按钮 → PUT 请求保存到服务器

### 灵感 API
- 新增 `POST /api/v1/story/inspire` 端点
- 前端「AI 给我灵感」按钮：弹出三栏建议弹窗（剧情×3、伏笔×3、角色×3）

| Bug | 原因 | 修复 |

#### v0.3.2：分层控制重构
废弃全章约束，改为"三层级"控制：

| 层级 | 内容 | 作用范围 | 约束力度 |
|:-----|:-----|:---------|:---------|
| **战术层（过渡）** | 上一章结尾 400 字 + 4 条过渡规则 | 仅控制新章前 200 字（第一段） | **超强约束** |
| **战略层（大纲）** | 完整大纲 + 3 条执行规则 | 从新章第二段开始至全章 | **最高优先级** |
| **自由层（风格）** | 上一章结尾 1500 字完整参考 | 贯穿全章 | **弱参考**，不强制复现 |

## v0.2.x — 基础功能

> 日期：2026-07-09

### 核心功能
- **作品管理**：作品的创建、读取、更新、删除（CRUD）
- **单篇故事生成**：基于多智能体协作的完整故事生成
- **角色管理**：角色的创建、编辑、删除，关联到作品
- **伏笔管理**：伏笔的创建、关联、追踪
- **历史记录**：生成过的故事列表，支持搜索、筛选、翻页
- **收藏功能**：故事收藏/取消收藏
- **故事编辑**：在详情弹窗内编辑故事内容，字数自动重新计算
- **故事导出**：Markdown / 纯文本格式导出

### 智能体协作生成管道
```
用户输入 → Planner（策划） → Character（角色构建） → Writer（写作） → Editor（润色） → 最终输出
```
- 基于 SSE（Server-Sent Events）流式输出
- 每阶段智能体状态实时反馈（running/writing/done/error）
- 多智能体上下文共享

### 后端 API（基础）

|:-----|:-----|:-----|
| `/api/v1/health` | GET | 健康检查 |

| `/api/v1/config/api-key` | POST | API Key 配置 |
| `/story/save` | POST | 故事保存 |
| `/story/{id}` | GET | 故事查询 |
